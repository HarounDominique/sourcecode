## Bugs Discovered (All New)

### BUG-4 — CRITICAL: rename-class destroys classes in other packages with same simple name

**File:** `src/sourcecode/rename_refactor.py`
**Severity:** Critical

When two classes share the same simple name across packages (e.g. `com.alpha.UserService` and `com.beta.UserService`), renaming `UserService → AccountService` modifies **both** class declarations, not just the target.

`com/beta/UserService.java` gets its `class UserService` declaration renamed to `class AccountService` — a different class in a different package is silently destroyed.

Root cause: no package-qualified disambiguation. Text substitution runs on all files where the simple name appears, including other class declarations.

**Fix:** After detecting `old_file` (which identifies the target class and its package), restrict class-declaration renaming to only `old_file`. All other files should only have import/type-reference updates, filtered by package import context.

**Reproduction:**
```bash
# Two classes: com.alpha.UserService and com.beta.UserService in same repo
sourcecode rename-class /repo --from UserService --to AccountService --dry-run
# BUG: both class declarations renamed, not just com.alpha.UserService
```

### BUG-1 — HIGH: find_java_files false positive on `/test/` substring

**File:** `src/sourcecode/repository_ir.py:3398`
**Severity:** High

```python
if "/test/" in rel or "/tests/" in rel or rel.startswith("test/"):
```

Substring match `/test/` hits `com/test/`, `org.context.test.*`, any package path containing the word "test" as a directory. Confirmed impact on BroadleafCommerce: 4 production files in `integration/src/main/java/org/broadleafcommerce/core/workflow/state/test/` are silently excluded.

Any user with `com.test` package gets `java_files_found: 0` — total analysis failure.

**Fix:**
```python
# Use path-segment check, not substring
parts = rel.split("/")
test_dir_indices = {i for i, p in enumerate(parts[:-1]) if p in ("test", "tests")}
# Only skip if test dir is in structural position (e.g. src/test/java, not src/main/java/com/test)
is_structural_test = any(
    i > 0 and parts[i-1] in ("src", "main") is False
    for i in test_dir_indices
)
# Simpler: check src/test/ prefix specifically
if "/src/test/" in rel or rel.startswith("test/") or rel.startswith("tests/"):
    continue
```

### BUG-3 — HIGH: @EnableMethodSecurity suppresses all SEC-001 findings

**File:** `src/sourcecode/repository_ir.py:203`
**Severity:** High

```python
_FILTER_SECURITY_ANNOTATIONS = frozenset({
    "@EnableWebSecurity",
    "@EnableMethodSecurity",    # ← wrong classification
    "@EnableGlobalMethodSecurity",
})
```

`@EnableMethodSecurity` enables per-method annotation security (`@PreAuthorize`/`@Secured`). It is NOT a filter chain. Classifying it as `filter_based` means any modern Spring Security config with `@EnableMethodSecurity` + `@PreAuthorize` → `security_model = "mixed"` → SEC-001 never fires.

**Fix:** Separate `@EnableMethodSecurity` from `_FILTER_SECURITY_ANNOTATIONS`. It should contribute to `_has_annotation_security`, not `_filter_based`. Keep `@EnableWebSecurity` (which configures the filter chain via `SecurityFilterChain`) as filter-based.

### BUG-6 — HIGH: cold-start --compact returns 89 tokens (schema mismatch)

**File:** `src/sourcecode/cli.py:5400`
**Severity:** High

Compact filter uses keys that don't exist in RIS output:
```python
_cs_keys = {"status", "git_head", "stacks", "entry_points",
            "key_dependencies", "project_type", "project_summary", "validation", "_meta"}
```
RIS actual keys: `summary` (not `stacks`), `entrypoints` (not `entry_points`), no `key_dependencies` or `project_summary` at top level.

Result: only 5 keys survive → 89 tokens instead of advertised ~10K.

**Fix:**
```python
_cs_keys = {"status", "git_head", "summary", "entrypoints", "endpoints",
            "project_type", "validation", "_meta"}
# Then truncate endpoints list to first 30 to stay ≤10K
```
