## Key technical context

### Cache directory layout
```
~/.sourcecode/cache/<repo_id>/
  core-<sha8>-<flags8>.json.gz      ← L1
  view-<corehash16>-<viewflags8>.json.gz  ← L2
  snapshot-<sha>-<flags>.json.gz    ← legacy
  cas/<sha256>.gz                   ← CAS blobs
```
`repo_id` = SHA-256[:16] of `str(canonical_repo_path)`.

### GC env vars
| Var | Default | Purpose |
|-----|---------|---------|
| `SOURCECODE_CACHE_KEEP_COMMITS` | 5 | Max distinct git commits to retain |
| `SOURCECODE_CACHE_MAX_CORES` | 20 | Core count cap (LRU eviction) |
| `SOURCECODE_CACHE_MAX_SIZE_MB` | 50 | Total size cap (oldest evicted first) |

### Async cache write
Background daemon thread writes L1+L2 after main output is printed. `atexit` join with 5s timeout. Thread starts only when `not no_cache and _core_key and not _pipeline_error`.

### Sandbox restriction (Claude Code shell)
Claude Code's shell has EPERM on renaming over existing workspace files (`os.replace()` blocked). Writing NEW files works. Workaround: `unlink() + write() + rename()` or run patch scripts from a normal Terminal.
